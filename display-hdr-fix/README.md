# display-hdr-fix

Make the OLED panel on the ASUS ExpertBook Ultra B9406CAA report its HDR
capability to KDE Plasma (and through KWin to Vivaldi / YouTube), so HDR
content actually plays in HDR instead of being tone-mapped down to SDR.

## The bug

The ASUS B9406CAA ships a Samsung Display **ATNA40LE01-0** OLED panel.
The full EDID (decoded with `edid-decode`) reveals a serious display:

```
Panel:    Samsung Display ATNA40LE01-0
Native:   2880x1800 @ 120 Hz
Pixel:    12 bpc native, 10 bpc supported in pipeline
Gamut:    DCI-P3 (Rxy=0.679/0.319, Gxy=0.240/0.713, Bxy=0.140/0.045)
Min lum:  4 cd/m²
Max lum:  700 cd/m² (full coverage), 1600 cd/m² (10% rectangular peak)
EOTF:     2.20 (SDR) and BT.2020 / SMPTE ST 2084 (HDR PQ)
Refresh:  30-120 Hz adaptive sync (VRR)
```

That's a DisplayHDR True Black 1000 class panel with full Wayland HDR
pipeline support already in KWin (Plasma 6.x exposes `wp_color_manager_v1`
with `set_luminances` + PQ ST2084 transfer function).

But:

```
$ kscreen-doctor -o | grep -E 'HDR|Wide'
    HDR: incapable
    Wide Color Gamut: incapable

$ stat -c%s /sys/class/drm/card0-eDP-1/edid
0
```

The `xe` driver reads **0 bytes** from the panel via DDC/AUX. Without an
EDID, the kernel hands KDE no panel metadata, so KDE marks the display
SDR-only and refuses to enable HDR. Every part of the pipeline downgrades:

- YouTube AV1-HDR videos in Vivaldi tone-map down to SDR (Rec.709 surface)
- 10-bit deep colour disabled, 8-bit RGB only
- DCI-P3 wide gamut unused (sRGB clipping, hugely undersaturated reds/greens)
- Reference white pinned at 200 cd/m² instead of 700+ cd/m²

The `xe` driver / Panther Lake combo ships in kernel 6.18+ with several
known eDP DDC edge cases. Until the AUX-channel EDID readback path is
fixed upstream, we work around it.

## The fix

Asus thoughtfully stashes the panel's full EDID — including the
DisplayID 2.0 extension block with all the HDR specs above — in an EFI
variable named `AsusEDID-607005d5-3f75-4b2e-98f0-85ba66797a3e` (256
bytes after the 4-byte EFI attribute prefix).

This module ships a one-time extraction of that EDID and tells the
kernel to use it instead of trying (and failing) to read DDC:

| File | Path | Why |
|---|---|---|
| `asus-b9406-edid.bin` | `/usr/lib/firmware/edid/` | The 256-byte EDID payload (base block + DisplayID 2.0 extension). SHA256 is verified against the live EFI variable on each install — if a BIOS update changes the panel EDID, we warn loudly so users know to file an issue. |

Plus a kernel cmdline addition to `/etc/default/limine`, regenerated by
`limine-update`:

```
drm.edid_firmware=eDP-1:edid/asus-b9406-edid.bin
```

This tells DRM "for connector eDP-1, use this firmware file as the EDID."
Same approach as `display-fix`'s `xe.enable_psr=0` — has to be on the
cmdline, not just `/etc/modprobe.d`, because xe loads from initramfs.

## Install

```sh
./patch.sh install display-hdr-fix
sudo reboot
```

After reboot, verify:

```sh
./patch.sh status display-hdr-fix
```

Expected output:

```
  cmdline:    drm.edid_firmware active in current boot
  EDID sysfs: 256 bytes (xe driver sees panel)
  KDE HDR:    capable
  KDE WCG:    capable
```

Then in System Settings → Display & Monitor → toggle **Enable HDR** on
the eDP-1 panel. From that point:

1. KDE compositor switches to PQ-encoded scanout
2. SDR apps stay tone-mapped correctly via `Allow EDR`
3. HDR-aware apps (Vivaldi, mpv with VO=gpu-next, MPV-HDR builds) get a
   real HDR surface
4. YouTube AV1-HDR videos play with the full PQ curve, peak highlights
   at 700-1600 cd/m² instead of clipped at 200

## Vivaldi / Chromium HDR notes

Vivaldi snapshot 7.10+ inherits Chromium 148+'s Wayland HDR video path,
but it isn't on by default. With this module installed and KDE HDR
toggled on, open `vivaldi://flags` and set:

- `Enable HDR Mode` → **Enabled**
- `Vulkan` → **Enabled**  *(better HDR colour management on Linux)*

Restart Vivaldi. YouTube settings → "Statistics for nerds" should now
show HDR colour primaries (`bt2020`) and EOTF (`smpte2084`) for HDR
videos. Without this module, the same statistics show `bt709` /
`gamma22` because the surface is SDR.

## Uninstall

```sh
./patch.sh uninstall display-hdr-fix
sudo reboot
```

Removes both the firmware file and the cmdline marker, regenerates the
limine config. After the reboot, the panel reverts to SDR (because xe
still can't read DDC EDID without our help).

## Why not patch the kernel xe driver?

Long-term the right fix is upstream — the xe driver should fall back to
the BIOS-provided ACPI `_DDC` method or the `_DSM` panel-discovery method
when AUX-channel EDID readback returns zero bytes. We're drafting that
patch separately (`upstream-patches/0004-...`), but until it's reviewed
+ merged + lands in stable kernels (~3-6 months), this firmware override
gives the same result with zero kernel maintenance.
